# DLoF Debugger (dlof-dbg)

[![License: MIT](https://img.shields.io/badge/License-MIT-yellow.svg)](LICENSE)
[![Build APK](https://img.shields.io/badge/Android-Auto--build-3DDC84.svg)](app/)

**مصحح أخطاء لصيغة DLoF (Document Loop Format) ولحزم `.dlofpkg`.**
مشروع Android مستقل بذاته (Kotlin + XML)، مبني من الصفر، بمستودعه الخاص، غير معتمد على مشروع القارئ `dlof-go`.

يفتح التطبيق ملف `.dlof` واحد أو حزمة `.dlofpkg` (أرشيف zip يحوي عدة ملفات `.dlof` مرتبطة في حلقة)، يفحصها محلياً بالكامل (بدون إنترنت)، ويعرض قائمة بالأخطاء والتحذيرات مع شرح كل مشكلة، ثم يتيح **تصحيحاً تلقائياً** للمشاكل الآمنة للإصلاح وحفظ نسخة مصححة.

## لماذا هذه الأداة؟

ملفات `.dlof` تعتمد على روابط ذاتية (`next`/`previous`) بدل فهرس مركزي — وهذا يعني أن أخطاء بسيطة (رابط لا يشير لملف موجود، رابط عكسي غير متطابق، معرّف مكرر، نوع محتوى غير معروف) يصعب اكتشافها بالعين المجردة، خصوصاً داخل حزمة تضم عشرات الملفات. **dlof-dbg** يفحص كل هذا آلياً في ثوانٍ.

## ماذا يفحص؟

| الفحص | الوصف | قابل للتصحيح التلقائي |
|---|---|---|
| صحة بنية XML | ملف غير صالح كـ XML (وسم غير مغلق، رمز غير صالح...) | لا |
| العنصر الجذر | وجود `<documentLoop>` بخاصيتي `id` و `version` | جزئياً (`version`) |
| `metadata` مفقود | كل ملف يجب أن يحوي `<metadata>` | لا |
| `content` مفقود | كل ملف يجب أن يحوي `<content>` بخاصية `type` معروفة | لا |
| `domain` غير معروف | قيمة المجال يجب أن تكون من القيم المعتمدة | لا (تحذير فقط) |
| `type` محتوى غير معروف | نوع المحتوى يجب أن يكون من الأنواع السبعة المعتمدة | لا |
| معرّف (`id`) مكرر | ملفان في نفس الحزمة يحملان نفس `id` | لا |
| رابط `next`/`previous` معطوب | يشير إلى `id` غير موجود في الحزمة | لا |
| رابط `next`/`previous` غير متطابق | `A.next = B` لكن `B.previous ≠ A` | نعم |
| ملفات معزولة | ملف داخل حزمة متعددة الملفات لا يرتبط بأي حلقة | لا (تحذير فقط) |

القواعد أعلاه مبنية على وصف الصيغة الوارد في توثيق DLoF العام (بنية `documentLoop`/`metadata`/`loopLinks`/`content` والمجالات وأنواع المحتوى السبعة). هذا المشروع **لا يتضمن ملف XSD رسمي** من مستودع DLoF الأصلي؛ المحرك هنا مكتوب من الصفر بناءً على تلك القواعد، لذا قد تحتاج لتعديل `validator/DlofRules.kt` إذا تغيّر المخطط الرسمي مستقبلاً.

## طريقة العمل

1. زر **"فتح ملف"** يفتح منتقي ملفات النظام (SAF) لاختيار `.dlof` أو `.dlofpkg`.
2. التطبيق يفحص الملف/الحزمة بالكامل محلياً (لا اتصال إنترنت، لا صلاحيات تخزين تقليدية).
3. تظهر قائمة بالمشاكل مرتبة حسب الخطورة (خطأ / تحذير) مع اسم الملف والعنصر المسبب ورقم السطر عند توفره.
4. إن وُجدت مشاكل قابلة للتصحيح التلقائي، يظهر زر **"تصحيح تلقائي وحفظ نسخة"** الذي يطبق الإصلاحات ويحفظ الملف المصحح في موقع تختاره.

## البناء محلياً

يتطلب JDK 17 و Android SDK (compileSdk 34). لا حاجة لأي مفاتيح توقيع لبناء نسخة `debug`:

```bash
./gradlew assembleDebug
# الملف الناتج: app/build/outputs/apk/debug/app-debug.apk
```

إن لم يكن `gradlew` موجوداً بعد الاستنساخ، يتولى GitHub Action توليده تلقائياً (راجع `.github/workflows/build-apk.yml`)، أو يمكنك توليده محلياً بـ:

```bash
gradle wrapper --gradle-version 8.7
```

## البناء عبر GitHub Actions

كل دفعة (push) على الفرع `main` تُشغّل تلقائياً بناء APK (نسخة debug قابلة للتثبيت المباشر) وترفعها كـ Artifact في تبويب **Actions**. لا حاجة لأي أسرار (Secrets) — هذا المشروع لا يتضمن نسخة موقّعة (release) ولا أي مفاتيح توقيع مضمّنة.
